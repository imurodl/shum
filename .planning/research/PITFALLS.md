# Pitfalls Research

**Domain:** Self-hosted Linux and Docker Compose upgrade, backup, rollback, and recovery tooling
**Researched:** 2026-03-11
**Confidence:** HIGH

> Proposed phases used below because `.planning/PROJECT.md` defines requirements but no roadmap yet:
> Phase 1 - Discovery and canonical state capture
> Phase 2 - Preflight checks and deterministic upgrade planning
> Phase 3 - Backup and snapshot orchestration
> Phase 4 - Upgrade execution, health verification, and rollback
> Phase 5 - Recovery drills, run history, and operator docs

## Critical Pitfalls

### Pitfall 1: Incomplete Compose Context Discovery

**What goes wrong:**
The tool upgrades the wrong effective stack, misses services, or recreates a different application model than the operator actually runs because it only reads a nearby `compose.yaml` and ignores project name, active profiles, `.env` interpolation, and the rendered Compose model.

**Why it happens:**
Docker Compose state is not defined by a single file path. Effective configuration depends on project-name precedence, profiles, environment interpolation, and the canonical model generated by `docker compose config`. Greenfield tools often discover by directory convention first and only later learn that real operators use `-p`, profiles, and per-host env files.

**How to avoid:**
Discover projects from the target host's live Compose context, not from repository assumptions. Capture and store the rendered `docker compose config`, resolved project name, active profiles, declared variables, service list, networks, and volumes before every run. Refuse to operate when interpolation is unresolved or the rendered model changes unexpectedly between discovery and execution.

**Warning signs:**
The CLI and the tool disagree on which services belong to a project, services appear only when profiles are enabled, operators use custom `-p` names, or the rendered config changes when `.env` changes.

**Phase to address:**
Phase 1 - Discovery and canonical state capture

**Primary sources:**
Docker docs for project names, Compose profiles, variable interpolation, and `docker compose config`.

### Pitfall 2: Non-deterministic Upgrades From Mutable Tags

**What goes wrong:**
An "upgrade" or "rollback" does not reproduce the prior state because the tool stores tags such as `latest` or `1.2` instead of immutable digests. The same run plan can produce different images on different days.

**Why it happens:**
Docker tags are mutable, while digests are immutable. Compose will recreate containers when configuration or image content changes, so a tool that records only tag strings cannot guarantee repeatability or safe rollback.

**How to avoid:**
Resolve and persist image digests before any change. Store the pre-upgrade container image IDs, rendered Compose config hash, and the exact target digests to be pulled. Use `docker compose config --resolve-image-digests`, `--lock-image-digests`, and config hashing as first-class planning primitives.

**Warning signs:**
Stacks use floating tags, the plan does not show digests, rerunning discovery yields a different target state without source changes, or rollback records only image tags.

**Phase to address:**
Phase 2 - Preflight checks and deterministic upgrade planning

**Primary sources:**
Docker docs for image digests, `docker compose config`, and `docker compose up`.

### Pitfall 3: Treating "Running" As Healthy

**What goes wrong:**
The tool reports success because containers are running, but the application is not actually ready. Dependent services race their databases, migrations finish late, or health flaps start after the tool has already declared victory.

**Why it happens:**
Compose starts and stops containers in dependency order, but official Docker guidance is explicit that startup order alone does not mean application readiness. Compose only waits for health when health checks and `depends_on: condition: service_healthy` are configured, and many greenfield tools stop at container state.

**How to avoid:**
Require or strongly guide operators to define health checks for stateful dependencies and user-facing services. Support post-upgrade verification using Compose health, HTTP/TCP probes, command checks, and bounded wait policies. Make "healthy" or "ready" the success boundary, not "container created" or "container running."

**Warning signs:**
Containers are `running` while logs show connection refusals, restart counts increase after the tool exits, or upgrades only pass when operators manually wait and retry.

**Phase to address:**
Phase 4 - Upgrade execution, health verification, and rollback

**Primary sources:**
Docker docs for Compose startup order and `docker compose up --wait`.

### Pitfall 4: Volume Copies That Are Not Recoverable Backups

**What goes wrong:**
The tool creates a tarball of Docker volumes or a local filesystem snapshot and calls it a backup, but restoring that artifact yields corrupt or incomplete state for live databases, or the backup disappears with the host because it never left the same storage stack.

**Why it happens:**
Official Docker volume docs show how to back up volume contents, but not every live application can be safely copied at arbitrary moments. PostgreSQL's official docs are explicit that filesystem-level backups require either a shut down server or a snapshot method supported by the database. Snapshot tooling such as Btrfs send/receive also requires deliberate snapshot handling and a separate destination. Inference from those sources: local snapshots are useful rollback aids, but they are not by themselves disaster-recovery backups.

**How to avoid:**
Support backup hooks that can quiesce applications, flush state, or call database-native backup flows. Classify artifacts as crash-consistent, application-consistent, or disaster-recovery-ready. Require off-host replication for anything labeled as a recovery backup, and record the backup method and artifact location with the run.

**Warning signs:**
The plan only copies `/var/lib/docker/volumes`, stateful services have no backup hooks, there is no off-host destination, or operators cannot say whether a backup is crash-consistent or application-consistent.

**Phase to address:**
Phase 3 - Backup and snapshot orchestration

**Primary sources:**
Docker docs for volume backup/restore, PostgreSQL docs for filesystem-level backups, and Btrfs send/receive docs.

### Pitfall 5: Image Rollback Assumes State Can Go Backward

**What goes wrong:**
Rollback swaps the previous image back in, but the service still fails because the upgrade already changed schemas, migration state, or on-disk formats. The old image no longer matches the data it is being asked to read.

**Why it happens:**
Rollback is often modeled as "old image in, problem solved." That is only true when application state stayed compatible. PostgreSQL's official upgrade docs treat major-version data migration as a dedicated process such as `pg_upgrade` or dump/restore, which is exactly the kind of state boundary that breaks naive image-only rollback.

**How to avoid:**
Detect stateful services and classify upgrades by reversibility before execution. Treat database major-version changes, migration jobs, and app-managed schema changes as special plans requiring a pre-migration backup ID and an explicit rollback strategy. Surface "image rollback only" versus "state restore required" before the operator approves the run.

**Warning signs:**
The target plan changes a database major version, includes a migration container or startup migration script, release notes mention one-way migrations, or rollback metadata does not include a pre-migration backup artifact.

**Phase to address:**
Phase 4 - Upgrade execution, health verification, and rollback

**Primary sources:**
PostgreSQL docs for `pg_upgrade` and filesystem-level backups.

### Pitfall 6: Destructive Cleanup In Error Paths

**What goes wrong:**
A failure handler tries to "get back to clean state" by using `docker compose down -v`, `rm -v`, `--renew-anon-volumes`, or broad orphan cleanup. The result is data loss or silent replacement of volumes that should have been preserved.

**Why it happens:**
Destructive flags are convenient during local development, and greenfield tools often inherit that ergonomics into production automation. Docker's official docs are explicit that `down -v` removes named volumes and that anonymous volumes are handled differently from named ones, but tools still blur "cleanup" and "recovery."

**How to avoid:**
Make destructive cleanup opt-in, loudly previewed, and unavailable on stateful services by default. Separate stateless cleanup operations from rollback operations. Inventory named, anonymous, and external volumes before the run and attach a destruction policy to each.

**Warning signs:**
Run plans include `down -v`, `rm -v`, or `--renew-anon-volumes`; services depend on unnamed volumes; or failure recovery is described as "just recreate everything."

**Phase to address:**
Phase 2 - Preflight checks and deterministic upgrade planning

**Primary sources:**
Docker docs for `docker compose down`, `docker compose up`, and Compose volume definitions.

### Pitfall 7: Missing Preflight For External Dependencies

**What goes wrong:**
The tool starts mutating containers before proving that the registry, external volumes/networks, backup target, or required permissions are available. Pulls fail with rate limits or auth errors, external resources are missing, and the host ends in a partially changed state.

**Why it happens:**
Greenfield upgrade tools often treat the host as if it were self-contained. In practice, Compose apps depend on registries, external resources, and backup destinations. Docker Hub documents pull limits and `429` failures, while Compose volume definitions explicitly distinguish `external` resources that the tool must not assume it manages.

**How to avoid:**
Run preflight checks for registry auth and image availability, external volume/network existence, backup destination reachability, free space policy, and execution permissions. Refuse to begin the mutating phase until all invariants pass, and provide a dry-run report that lists every external dependency the run assumes.

**Warning signs:**
Images are pulled anonymously from Docker Hub, the stack declares `external: true` volumes or networks, operators rely on remote backup repositories, or prior failures include `429 Too Many Requests` and permission errors.

**Phase to address:**
Phase 2 - Preflight checks and deterministic upgrade planning

**Primary sources:**
Docker Hub usage-limit docs and Docker Compose volume/network docs.

### Pitfall 8: Backups Exist, But Recovery Is Untested

**What goes wrong:**
The operator has backup artifacts and maybe even repository integrity checks, but an actual restore is slow, incomplete, or impossible because the tool never practiced restoring Compose config, secrets references, volumes, and service order on a clean host.

**Why it happens:**
Backup creation is easier to automate than recovery rehearsal, so tools stop at "artifact created." restic's official docs separate repository checking from restore operations, which is a useful reminder that integrity verification and usable service recovery are different problems.

**How to avoid:**
Build restore drills into the product. Capture the rendered Compose model, project name, image digests, backup artifact IDs, secret references, and a recovery order that can be replayed on an isolated host. Track last successful drill time, measured RTO/RPO, and whether the recovery used only documented steps.

**Warning signs:**
There is no recent restore-drill timestamp, the backup system only runs integrity checks, recovery requires undocumented tribal knowledge, or operators cannot state how long recovery takes.

**Phase to address:**
Phase 5 - Recovery drills, run history, and operator docs

**Primary sources:**
restic docs for repository checking and restore operations, plus Docker docs for Compose config and project naming.

## Technical Debt Patterns

Shortcuts that seem reasonable but create long-term problems.

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Store image tags but not digests | Faster MVP and simpler UI | No deterministic rollback, weak auditability, hard-to-reproduce bugs | Never |
| Discover stacks by scanning for `compose.yaml` only | Quick host discovery demo | Wrong project targeting when profiles, custom project names, or env interpolation are used | Only in a throwaway prototype with no execution capability |
| Treat every backup as a generic volume tarball | Easy to implement across apps | Corrupt restores for live databases and false confidence in disaster recovery | Only for explicitly stateless data |
| Use `down -v` or volume renewal as a generic rollback primitive | Simple reset behavior | Turns recoverable incidents into data-loss incidents | Never |
| Skip restore drills until after launch | Saves early engineering time | Backup system remains unproven when the first incident happens | Only before any external users or production-like claims; not acceptable for v1 |
| Keep run history only on the managed host | Minimal storage design | Incident history disappears with host failure | Acceptable only if duplicated to an operator machine or external store in the same phase |

## Integration Gotchas

Common mistakes when connecting to external services.

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| Docker Compose projects | Assuming path name equals Compose project identity | Capture rendered config, project name, profiles, and variables from the target host before execution |
| Container registries | Pulling anonymously and discovering auth/rate limits during the maintenance window | Preflight registry auth, pullability, and target digests before any mutating step |
| External volumes and networks | Treating `external: true` resources as if the tool owns lifecycle and rollback | Model them as prerequisites, verify they exist, and do not delete or recreate them automatically |
| PostgreSQL and other stateful services | Backing up files live without app-aware hooks | Use database-native or quiesced backup hooks and store the artifact ID with the run |
| Snapshot-based backup tools | Calling same-host snapshots a recovery solution | Label snapshots as local rollback aids unless replicated to a separate failure domain |
| Backup repositories | Checking repository integrity but never rehearsing service restore | Pair integrity checks with isolated restore drills and measured recovery objectives |

## Performance Traps

Patterns that work at small scale but fail as usage grows.

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Serial image pulls and health waits over one SSH control path | Upgrade windows grow linearly with service count and image size | Pre-pull targets where safe, parallelize read-only checks, and use bounded concurrency for pulls | Multi-service hosts with several large images, typically 5+ services or slow uplinks |
| Full-volume backup before every run | I/O spikes, long freezes, and operators disabling backups under pressure | Support policy-based backups, snapshot hooks, and app-specific incremental tools | Stateful stacks with large databases or media volumes, often tens of GB and up |
| Recomputing discovery from scratch on every command | CLI feels slow and remote hosts get hammered by repeated inspect/config calls | Cache discovery results with explicit invalidation when Compose config hash changes | Hosts with many Compose projects or frequent operator polling |
| Fixed post-upgrade sleep timers | Either slow success paths or flaky false failures | Use service-specific health deadlines and stop as soon as readiness is proven | As soon as services have different startup characteristics |

## Security Mistakes

Domain-specific security issues beyond general web security.

| Mistake | Risk | Prevention |
|---------|------|------------|
| Logging rendered Compose config and environment values without redaction | Secrets leak into run history, support bundles, or CI logs | Redact known secret fields and avoid persisting raw env values unless explicitly requested |
| Disabling SSH host verification for convenience | The tool can connect to the wrong host or a spoofed endpoint | Require normal host-key verification and make exceptions explicit and auditable |
| Running all remote actions as unrestricted root | A product bug becomes a host-compromise event | Prefer least privilege, narrow sudo rules, and explicit capability checks |
| Storing backup-repository credentials in plain text on the target host | Backup system becomes an easy lateral-movement target | Use dedicated credentials, secret storage, and minimal-scope tokens |
| Pulling and rolling back by mutable tags only | Supply-chain ambiguity and unreviewable production state | Pin digests, record provenance where possible, and display the exact digest being deployed |

## UX Pitfalls

Common user experience mistakes in this domain.

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| "Upgrade succeeded" means only "containers started" | Operators trust a broken deploy and discover failure later | Make readiness and verification the success boundary |
| No preview of exact digests, volumes, and external resources affected | Operators approve risky changes without understanding blast radius | Show a dry-run plan with changed images, config hash, volume policy, and rollback boundary |
| Rollback button without migration-risk warning | Users expect reversibility where none exists | Present rollback class explicitly: image-only, image-plus-restore, or manual recovery required |
| Backup status shown without restore-drill status | Users infer recoverability from artifact presence | Show last successful drill, measured recovery time, and artifact lineage next to backup status |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **Discovery:** Often missing project-name, profile, and `.env` resolution - verify the tool stores the rendered `docker compose config` and canonical project identity
- [ ] **Upgrade planning:** Often missing immutable digests - verify every plan records current and target digests plus a config hash
- [ ] **Health verification:** Often missing real readiness checks - verify success requires health/probe completion, not only `running`
- [ ] **Backup:** Often missing app-consistency classification - verify every stateful service has either a quiesce/native-backup hook or an explicit "unsupported" warning
- [ ] **Rollback:** Often missing migration awareness - verify the plan distinguishes reversible image swaps from state-restore-required rollbacks
- [ ] **Recovery:** Often missing restore rehearsal - verify there is a recent isolated restore drill with documented RTO/RPO
- [ ] **Run history:** Often missing off-host retention - verify incident history survives loss of the managed host

## Recovery Strategies

When pitfalls occur despite prevention, how to recover.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Incomplete Compose context discovery | MEDIUM | Stop automation, capture the live rendered config and project identity from the host, compare against stored hashes, then rerun against the corrected target only |
| Non-deterministic upgrade from mutable tags | MEDIUM | Inspect currently running image IDs, resolve digests from registry and local cache, reconstruct the last known-good config, and pin digests before retrying |
| "Running" but not healthy | MEDIUM | Revert to recorded last known-good digests if state is unchanged, then add or fix health probes before another upgrade attempt |
| Non-recoverable volume backup | HIGH | Restore the last known application-consistent backup if available; if only crash-consistent artifacts exist, restore to an isolated host first and validate data before production cutover |
| Image rollback after state migration | HIGH | Restore pre-migration data from the captured backup/snapshot, deploy the prior image against that restored state, and treat the data rollback window as explicit operator impact |
| Destructive cleanup removed volumes | HIGH | Local rollback is usually impossible; recover from off-host backup, rebuild the Compose project, and review whether any data can be replayed from upstream systems |
| Missing preflight for external dependencies | LOW to MEDIUM | Freeze further changes, satisfy the missing registry/auth/external-resource prerequisites, then rerun from the stored preflight plan instead of guessing manually |
| Recovery never rehearsed | MEDIUM to HIGH | Pause feature work, run an isolated restore drill end to end, document the exact sequence and timings, then make drill status visible before resuming upgrades |

## Pitfall-to-Phase Mapping

How roadmap phases should address these pitfalls.

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| Incomplete Compose context discovery | Phase 1 | Tool can reproduce the same rendered Compose model, project name, profiles, and variables across repeated discovery runs |
| Non-deterministic upgrades from mutable tags | Phase 2 | Every plan and history record contains current and target digests plus a config hash |
| Treating "running" as healthy | Phase 4 | Integration tests prove that unhealthy services fail the run even when containers stay running |
| Volume copies that are not recoverable backups | Phase 3 | Backup metadata records consistency class, artifact location, and hook outcome for every stateful service |
| Image rollback assumes state can go backward | Phase 4 | Rollback planner blocks or escalates migration-bearing changes unless a pre-migration backup and rollback class are present |
| Destructive cleanup in error paths | Phase 2 | Executor rejects destructive volume flags by default and requires explicit policy overrides |
| Missing preflight for external dependencies | Phase 2 | Dry-run report fails when registry auth, external resources, or backup destinations are unavailable |
| Backups exist, but recovery is untested | Phase 5 | Product tracks last successful restore drill, measured recovery time, and whether recovery used only documented steps |
